# Stage 1 — 搜集计划的建法与模板

前提：已完成 Stage 0.5（读完所有 channels/*.md + log-format.md）。

## 铁律（不可违背）
- **8 渠道全覆盖，不许跳过任何渠道**。任何题目（消费选题类 / 开发者工具类 / 其它）在任何平台都可能有人发布相关内容——工具选型类不是"只去 GitHub"，抖音/小红书上也有剪映自动字幕、通义听悟这类真实工具信息。低相关 ≠ 跳过。
- **无"权重"列**。不分主次、不厚此薄彼；用**深度**分配抓取量（默认 15/渠道）。低相关渠道可由用户调低深度，但不排除、不降权。
- **每渠道多角度关键词**：至少 2–3 个不同角度的词（正面词 / 痛点词 / 对标词 / 英文同义词），不要单一视角。
- **要提取的信号**：逐渠道写清抓什么——爆款互动指标 / 开源方案成熟度(star·活跃·License) / 教程步骤 / 讨论区痛点 / 价格 / 准确度声称。
- **付费 ASR 必须单独知情授权**：计划中必须明确可能调用 fun-asr 等按量计费服务，按最终入选且无可用字幕的视频总时长给出**数值化估算上限**（最多 `<ASR_HOURS>` 小时、最多 `<ASR_CNY>` 元；当前参考价约 0.8 元/小时，以服务商实际计费为准）。用户必须明确同意该上限，批准搜集计划本身不自动授权付费；不得超过上限，预计会超出时须重新询问。未明确同意（含无交互运行）时不得调用付费 ASR，可取原生/免费字幕；必须 ASR 才能补全的视频只能申报 `capture.video.status=failed`，并写明“未获付费 ASR 授权”，不能静默跳过或伪装成功。
- **预计耗时如实展示**（用户要在知情下批准，不许让他盲批）：
  - 收集阶段：会开可见浏览器的抖音/小红书/知乎/B站顺序执行，避免同时弹窗和浏览器启动冲突；登录/验证码窗口只提示一次并最多等待 120 秒，无人响应就终止该渠道并逐关键词申报失败，不反复弹窗。YouTube/GitHub/Twitter/web 并行。搜索后再做一轮独立证据复核。视频只处理最终入选内容：抖音/小红书直传 fun-asr，B站优先 ai-zh 字幕、无字幕才下载转写，YouTube 优先原字幕，Twitter 所有带视频的入选推文均须字幕/ASR。社交内容抓前 10 条有用评论，小红书识别全部入选图文笔记配图（视频封面不强制），GitHub 核对实际 LICENSE 文件。Stage 2 典型 **20–45 分钟**，长视频/大量配图时更久。
  - 总结阶段：通读 + 生词建卡 + 定点核查，sub 帮手数量随生词与承重断言数走（典型 10–30 个，并行执行），典型 **10–30 分钟**。
  - 全程典型 **30–60 分钟**。用户时间紧可当场调低各渠道深度/关键词数换速度。

## 批准闸
明确等用户批准后，依次做两件事：
1. **回填 manifest**：把最终计划写入 `<OUT_DIR>/manifest.json` 的 `plan` 字段：`{"channels":[{name,keywords,signals,depth},…],"dimensions":[…]}`。**name 必须转成英文标准名**（douyin/xiaohongshu/zhihu/bilibili/youtube/github/twitter/web）——Stage 3 总结以 manifest 为准、不依赖回看对话（对话可能被压缩丢原话）；不回填，用户批准的对比维度就丢了。同时写入顶层 `asr_authorization:{"authorized":true|false,"max_hours":<数值>,"max_cost_cny":<数值>}`；只有用户明确同意展示过的上限才能写 `true`，未明确同意一律写 `authorized:false,max_hours:0,max_cost_cny:0`。
2. 用该计划对象（外加 skillDir/outDir 绝对路径）调 Stage 2 的 Workflow。
（无交互运行时：跳过等待，按默认计划执行并在 manifest 记 `"approved": false`、`"asr_authorization":{"authorized":false,"max_hours":0,"max_cost_cny":0}`。）
