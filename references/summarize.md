# 总结执行规程（Stage 3，主 agent 本人执行）

本规程的执行者是**主 agent 本人**——2026-07-15 用户反馈拍板：不再派发独立总结 agent（中转通信绕，且子 agent 进程退出会让对话续不上），与用户的全部沟通由主 agent 直接进行。你的职责：通读全部渠道笔记 → 建生词卡 → 核实关键点 → **与用户完成一轮沟通（必经）** → 产出可执行方案。建卡、核查这类单点活照旧派 sub agent 并行干（已验证可用），但通读、判断、综合、与用户对话不外包。

输入（`<OUT_DIR>`/`<SKILL_DIR>` 为本会话已知的绝对路径）：`<OUT_DIR>/manifest.json` + `<OUT_DIR>/raw/findings.*.jsonl`（全部渠道）+ `<OUT_DIR>/coverage.json`（证据完成/失败统计）+ Stage 2 校验产出的规模统计。

## 类比：像人做旅游攻略笔记

skill 作者的原话（🔒 一字不改，任何迭代禁止润色/概括/替换）：

> **笔记的目的是，尽可能涵盖这个文字帖/视频里作者的观点/建议/论断/流程/方案等精华，以后回顾时如无必要只看精华笔记，无需再看冗长的原帖。**

作者去京都前会各渠道看大量帖子，每帖把精华记在一张纸上，最后**摊开全部纸做综合决定**——三条笔记示范原文见 log-format.md《note 字段质量标准》（你已被要求完整读它）。你现在做的就是"摊开全部纸"这一步。两个关键差异要补偿：
1. **你没有人的背景知识**——遇到不认识的新名词（可能超出你训练数据），必须现场查、建卡，不许不懂装懂。
2. **技术数字不像旅游断言那样可信**——"贵船神社很远"没人骗你，但"错字率 7.81%"是厂商自评、有立场。承重的数字必须换个角度核。

## 读取协议（先看这节；禁止用 Read 直接读 findings 文件）

finding 是单行 JSON、行内嵌原文全文（可达数万字），Read 工具对超长行会**静默截断**。一律用投影脚本：

| 目的 | 命令 |
|---|---|
| 规模/覆盖预检（熔断判断） | `python3 <SKILL_DIR>/scripts/project_notes.py --raw-dir <OUT_DIR>/raw --mode stats` |
| 第一遍速览（全部 headline） | `… --mode headlines` |
| 第二遍通读（全部 note，不含 content） | `… --mode notes` |
| **反查原帖全文**（按 id 取整条，含 content+raw） | `… --mode get --id xhs-007` |

`--mode get` 就是"从笔记反查指定帖子原文"的正规通道——需要引用原文或笔记存疑时用它，不必回网上重搜。

## 规程（按序执行）

### 1. 速览后通读——一条不许漏
先读 coverage.json。只要出现 `missing` 状态，立即停止并按 SKILL.md Stage 2 的单渠道重派 SOP 重跑对应渠道的证据补全；不要用标题/简介替代缺失的视频、评论、图片或许可证处理。`failed`/`not_available` 可以继续，但必须在报告附录展示数量和原因。还要逐渠道检查 `query_coverage`、`failures`、`skipped`：每个批准关键词的命中数或失败/跳过原因都必须进入附录，不能只报渠道总条数。
先跑 `--mode stats` 定规模，再 `--mode headlines` 读全部一句话标题建立全景，最后 `--mode notes` **通读全部 note，逐条读完**。
- **熔断口径**：以 stats 输出的 `approx_tokens_conservative` 字段（headline+note 的保守 token 估算，content 不计入）为准。若超过约 5 万 token，**立即停止并向用户如实报数**，请用户在"拆成多次调研 / 缩小范围 / 明确授权一次性完整处理"中选择。**绝不静默取子集、绝不截断**——这是 v1 最大的病，不许重犯。
- 通读时手上开三张清单：**矛盾清单**、**承重断言清单**（第 3 步用）、**交叉印证清单**（第 5 步用）——凡是**不同渠道、或同一渠道不同作者**独立重复的同一论断（"X 好/X 不值/X 需要预约/X 大概多少钱/X 要赶早"），记下论断一句话 + 全部来源 finding id。同一作者跨平台发同一内容不算独立印证。

### 2. 生词建卡（治"落后几代不自知"）
汇总各条的 `unknown_terms` + 你自己读时不认识的名词。**每个词派 1 个 sub agent** 查证（一词一帮手，不设数量上限）。**派发词逐字使用下面模板**（判据必须照抄原文进 prompt，不许转述）：

    查证名词「<词>」：搜官网/发布公告/论文，查清"是什么/谁做的/何时发布/取代了谁"。
    只查这一个词，禁止扩大范围、禁止顺带调研别的。
    返回一行 JSON：{"type":"term","term":"<词>","what":"…","released":"YYYY-MM","by":"…","supersedes":"…或null","sources":["…"]}

结果写入 `<OUT_DIR>/verify/glossary.jsonl`。建完卡后**按 released 排出方法/模型时间线**——推荐任何方案前先看它站在时间线哪一代，别把老一代当最新推荐。

### 3. 标记冲突与承重断言
通读中记下两类点：
- **冲突**：笔记互相矛盾处（"A 说免费，B 说要 VIP"）。
- **承重断言**：你拟推荐的方案所依赖的关键数字/说法（准确率、价格、"X 比 Y 快 N 倍"、有无接口）。判据：**这句话若为假，结论就得改** → 承重，要核；改了也不影响选型（如某帖点赞数）→ 不核，省钱。

### 4. 定点核查（每点派 1 个 sub agent，不设数量上限）
**派发词逐字使用下面模板**（方法论必须照抄进 prompt，不许转述）：

    核查断言：「<断言原文>」（来源 finding：<id>）。
    按问题性质二选一，禁止用错方法：
    - 事实题（价格/授权/有无接口/是否停更）：官方页面即权威。查官网价格页、授权文件原文、产品文档，查到即裁决。
    - 品质题（准不准/快不快/好不好用）：官方自评是利益方陈述，"找到厂商原文"不算核实（只证明"官方这么说过"）。
      必须找独立信号：搜「<名字> 口碑/实际体验/踩坑/吐槽」、翻真实用户帖、看 GitHub issues 抱怨密度、找非利益方评测。
      独立信号不足以定论 → verdict 填 unverifiable，不许强行下结论。
    只查这一个问题，禁止递归发起全量调研。
    返回一行 JSON：{"type":"verdict","id":"vd-<3位序号>","claim":"…","finding_ids":["…"],"verdict":"confirmed|refuted|unverifiable","evidence":["…"]}

裁决写入 `<OUT_DIR>/verify/verdicts.jsonl`（id 形如 `vd-001`，报告/runbook 引用它）。报告里厂商自评数字一律标"厂商自评"；核不动的明标"未证实"；不足定论的进 runbook 的 `to_test` 留给动手实测。

### 5. 与用户对话（必经步骤，无论 refined/rough；先讲解「名词速览+交叉印证」、后选择题，qa.md 只许追加存档）
**出方案前必须与用户至少完成一轮沟通。** 用户的原始要求是："不管我有没有添加更多的约束，你都需要让最后的总结 agent 与我进行沟通"。maturity 只影响问多少：
- rough：2–4 个定向问题。
- refined：1–2 个，聚焦**调研中新浮现、用户当初未必意识到的约束**（如金钱预算/耗时/质量-成本取舍/上手难度/维护成本）。

问题的两类合法来源（不限于第一类）：
a) 笔记里的**真实分歧**，自带背景让用户看得懂能选。例："方案分两派：免费网页工具（手动上传、零部署）vs 本地脚本（能全自动）。你是偶尔手动转几条就够，还是要接进自动化流程？"
b) **调研中浮现的新约束维度**——未必表现为笔记分歧，但会改变推荐。例："所有像样的生成方案都要花钱：便宜的每分钟约 X 元、效果好的约 Y 元。你有预算上限吗？"

**沟通固定为两段式，顺序不可颠倒（2026-07-15 用户反馈拍板）。第一段先"讲"、第二段才"问"。用户是因为不了解这个领域才发起调研的——默认他不认识调研中出现的任何专有名词：**

**第一段·讲解 + 答疑。** qa.md 开头必须依次是以下两节，一并呈现：
1. 「名词/地点速览」：对即将出现在问题、选项或候选方案里的**每一个**专有名词（地名/景点/工具/型号/术语），各写 1–2 行——是什么 + 在哪/属于哪一类 + 与本次调研为什么相关（可带 finding id）。
2. 「交叉印证」（2026-07-15 用户反馈新增）：从通读时的交叉印证清单产出，逐条编号 C1/C2/…——每条 = 论断一句话（"X 好玩/X 不值/X 需要预约/X 约多少钱"）+ 印证来源（**≥2 个相互独立的 finding id**，跨渠道优先，同渠道不同作者也算；有官方核查的附 vd-id）。这些"从不同人嘴里拿到的相同论断"是方案的锚点，默认路径应尽量落在锚点上。**单一来源的说法不得混入本节**——要提可单独标注"仅单一来源，待证实"。
两节写入 qa.md 后原文呈现给用户，并明确问："以上有没有想先弄清楚的？"答疑到用户表示没有疑惑为止（每轮追问与解答同样原文追加进 qa.md）。⚠️呈现方式（2026-07-15 实录）：讲解正文用**纯文本消息**输出、该回合**不要**同时调用交互式问答控件（AskUserQuestion 等）——控件弹窗会把长正文挤出用户视野，用户会说"我没看到这两块"；确认无疑惑这一问用纯文本收尾，或放到正文完整送达之后的下一回合再问。
**第二段·选择题。** 用户确认无疑惑后，才呈现 Q1/Q2/…。出题自查：选项中出现的每个名词，要么已在速览里介绍过，要么行内括号短释；选项与某条交叉印证相关时直接引用其编号（如"据 C2"）；每题注明"你帮我定"时采用的默认值。禁止跳过第一段直接出选择题。

**存档保真（qa.md 是问答的唯一权威记录，只许追加；防上下文压缩丢原话）**：
1. 讲解两节与问题先**原文**写入 `<OUT_DIR>/qa.md`（「名词/地点速览」「交叉印证」两节 + Q1/Q2/…），再呈现给用户——禁止只在对话里问而不落盘。
2. 用户每次回答/追问，**立即一字不改**追加进 qa.md（标 A1/A2/…；追问按 Q 序号续排）。全程禁止总结/转述/改写/臆想/代答。

### 6. 交付（先落盘，再由主 agent 呈现）
按 `<SKILL_DIR>/references/report-format.md` 产出两份：
- `<OUT_DIR>/report.html`（给人审）：含**时间线**小节、方案区用**默认路径+切换条件**结构、附录含 verdicts（含"未证实"清单）+ glossary + 各渠道 meta 覆盖率 + qa.md 问答原文。
- 报告附录必须原样汇总 coverage.json：每渠道入选数、每个批准关键词的命中/失败/跳过、视频字幕/ASR成功与失败、评论条数、图片识别完成与失败、许可证 verified/unknown。禁止只写“已覆盖”。
- `<OUT_DIR>/runbook.json`（给 AI 执行）：default_plan 每步到命令/模型名级 + fallback 触发条件 + timeline + verified + unverified + **to_test** + open_questions。
- **每个结论必须带 finding id 或 verdict id（vd-xxx）引用**，无引用的结论不许出现。
落盘后按 SKILL.md《交付》一节呈现：程序化切片引用 summary/plans/reco 三节原文 + 给出两份文件路径，不整读全文、不凭记忆复述；对首次出现的关键名词附一行短释（取自 glossary）。

## 无交互（headless/ultracode）模式
没有人能回答第 5 步。规程照跑，唯第 5 步改为：把要问的问题（两类来源都要）写进 runbook 的 `open_questions`，每题附"若选 A 则默认方案 X、若选 B 则方案 Y"的分支，不阻塞、直接落盘全部产物。**无论 maturity 是什么都要留问题**（refined 只是问题更少更聚焦）。
